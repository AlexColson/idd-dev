## OVERVIEW

Temporary development scaffolding for the IDD (Intent-Driven Development) ecosystem. Provides a unified Cargo workspace root and Nix devshell so 3 component crates (`idd`, `idd-fir`, `idd-dsl`) can be built/tested together during bootstrap. Goes away once components are published to crates.io.

IDD is a process where **declared intent drives every phase of development** — from design through implementation through runtime observation. Constraints are the primary design artifact, not code. The system combats "drift" — the accumulation of invisible coupling and unstated assumptions that kills coherence in multi-agent development.

## COMPONENT ARCHITECTURE

| Component | What It Is | Crate Structure | Build Order |
|-----------|-----------|-----------------|-------------|
| `idd-dsl` | Foundation — grammar, AST, type system, `#[intent(...)]` proc-macro | `idd-dsl-core` → `idd-dsl-parser` → `idd-macros` | 1st |
| `idd-fir` | Feature intent registry — "mini Jira" in `.fir/` | `idd-fir-core` → `idd-fir-fs` → `idd-fir-cli` | 2nd |
| `idd` | Root binary — CLI + MCP server, wires all subcommands | `idd-core` → `idd-cli` | 3rd |
| `idd-obs` | Runtime observation — telemetry, traces, violations | (later, optional dep of `idd-macros`) | 4th |

### Dependency Graph

```
idd-dsl-core ←── idd-dsl-parser
      ↑
idd-macros ──(optional)──→ idd-obs
      ↑
  (dev-dep in annotated crates)

idd-fir-core ←── idd-fir-fs ←── idd-fir-cli
      ↑
idd-dsl-core (constraint IDs only)

idd-core ←── idd-cli
      ↑
idd-dsl-core, idd-fir-core, idd-obs
```

**Key invariant:** `idd-dsl-core` depends on nothing in this ecosystem. It is the foundation.

### Extension Protocol

Any binary named `idd-<name>` on `$PATH` is automatically a subcommand (cargo-plugin convention). `idd serve` exposes every subcommand as an MCP tool — agents and humans hit identical code paths.

## CONVENTIONS

- **Submodule lifecycle**: Skeleton first → populate → publish to crates.io → remove from this workspace
- **Nix modules**: All `nix/modules/*.nix` auto-imported via `builtins.readDir` in `flake.nix`
- **Pre-commit config**: Auto-generated by git-hooks.nix — edits to `.pre-commit-config.yaml` are lost on rebuild
- **Workspace members**: Commented out by default; uncomment as submodules are initialized
- **Each repo structure**: `Cargo.toml` (workspace root) → `crates/` (individual publishable crates) → `docs/` (design docs, migrate to `.fir/` later)
- **Dogfooding**: `idd-fir` tracks its own features in its own `.fir/`; `idd check` runs in CI for every repo

## ANTI-PATTERNS (THIS PROJECT)

- **NEVER** edit `.pre-commit-config.yaml` directly — modify `nix/modules/pre-commit.nix`
- **NEVER** commit `[patch.crates-io]` entries to submodules — they live only in `.cargo/config.toml` here
- **NEVER** publish this workspace to crates.io — it's dev-only scaffolding
- **NEVER** let `idd-dsl-core` depend on `idd-macros` — dependency arrow is one-way
- On publishing: **remove** this `Cargo.toml` entirely; each component declares its own deps
- Extensions are **never** submodules of `idd-dev` — they're third-party repos
